## Architecture Completed (2026-07-03, read before implementation)

Full spine finalized at `_bmad-output/planning-artifacts/architecture/architecture-bmad-expense-tracker-2026-07-03/` (`ARCHITECTURE-SPINE.md` — 10 `AD`s, the enforceable contract — plus `SOLUTION-DESIGN.md`, a concise human-readable companion). Went through a 6-agent reconciliation + reviewer gate against the PRD, UX spec, and this file. Key non-obvious rules agents should already know, not re-derive:

- **Layered monolith, split hosting:** `apps/web` (Next.js, Vercel) talks to `apps/api` (Spring Boot, Railway) via REST/JSON only; `apps/api` alone owns PostgreSQL. Monorepo: `apps/web` + `apps/api`. Controller → Service → Repository even for trivial endpoints; DTOs at the API boundary always, never a raw JPA entity.
- **Category has three kinds, not two:** `DEFAULT` (5 seeded, renameable/re-iconable, never deletable), `CUSTOM` (user-created, fully editable/deletable), `SYSTEM` (Uncategorized — a real seeded singleton row, not a `NULL` sentinel). `CategoryService` itself rejects any create/rename/delete against the SYSTEM row — not just hidden from the UI.
- **Budget is per-Period and auto-copies forward.** Resolves an ambiguity the PRD left open: spend resets to ₹0 every Period (confirmed, no rollover), but the *budget figure* carries forward from the prior Period unless never set. Practical effect: the "set your first budget" neutral prompt (PRD UJ-4) now only ever appears once, not every month — PRD `addendum.md` was updated to match.
- **Transactions are immutable once saved** — no update/delete endpoint exists in MVP scope (`POST`/`GET` only). This was implicit in the PRD Glossary but wasn't an enforced architectural rule until now.
- **Asia/Kolkata is the only clock that matters.** All "today"/Period-boundary logic is computed server-side, fixed to IST — the client never computes a date boundary itself. Closes a real bug class (server-UTC vs. browser-IST disagreeing near midnight).
- **IDs are auto-increment `BIGINT`**, not UUID. **Money is always `NUMERIC(12,2)`/`BigDecimal`**, never float — client only ever displays server-computed totals, including the Search & Filter running total.
- **Testing:** backend `@WebMvcTest` + direct unit tests on the derived-totals/budget-status calculation specifically; frontend component tests (Vitest/Testing Library) on Quick Add and Budget Status. No e2e for MVP. **CI:** GitHub Actions gates every push/PR (tests+lint must pass before merge).

## Story 1.1 Implemented (2026-07-05, read before touching `apps/web`, `apps/api`, or any later story)

Full monorepo scaffold now exists and is verified live (not just written): `docker compose up` → healthy Postgres 18.4; Spring Boot boots and `/actuator/health` → `200`; `GET /api/categories` returns the seeded 5 `DEFAULT` + 1 `SYSTEM` rows as DTOs; restart-idempotency confirmed (no duplicate rows on a second boot); CORS allows `http://localhost:3000` and rejects other origins (403); `mvn test` 8/8 pass; `npm run lint`/`npm run build` pass. Full detail in the story file's Dev Agent Record — key things later stories/agents must know:

- **Schema-management decided:** Hibernate `spring.jpa.hibernate.ddl-auto=update` for table/column creation. The two AD-5 indexes (SYSTEM-row partial unique index, case-insensitive `lower(name)` functional unique index) are applied as native SQL via `JdbcTemplate`, run after context startup — **follow this same pattern** (ddl-auto + `JdbcTemplate` for anything ddl-auto can't express) for `transactions` (Story 1.2) and `budgets` (Story 1.5) unless a deliberate decision is made to introduce a migration tool instead.
- **AD-4 precedent set during Story 1.1's code review: "only a Service may call a Repository" applies even to one-time startup/bootstrap code, no exception.** `CategoryBootstrapRunner` (a `@Component implements CommandLineRunner`) originally called `CategoryRepository`/`JdbcTemplate` directly for seeding; a code-review finding flagged this as a literal AD-4 violation, and the user explicitly chose strict compliance over documenting an exception. The fix: `CategoryService` gained `ensureSchemaHardened()`/`ensureDefaultCategoryExists()`/`ensureSystemCategoryExists()`, and `CategoryBootstrapRunner` now only calls `CategoryService` — it has zero direct `CategoryRepository`/`JdbcTemplate` access. **Any future seeder (`transactions`/`budgets` or otherwise) must follow this same shape**: the `CommandLineRunner` orchestrates, the `Service` owns every Repository/JdbcTemplate call, including native-SQL schema hardening.
- **Idempotent seeding must also guard against concurrent-instance races, not just repeated sequential runs.** The original seeder only had a check-then-act guard (`findByNameIgnoreCase(...).isEmpty()` then `save()`), which is safe for sequential re-runs (verified by a test) but not for two instances starting concurrently — a real gap caught by code review. `CategoryService`'s `ensureDefaultCategoryExists`/`ensureSystemCategoryExists` now wrap the `save()` call in `try/catch (DataIntegrityViolationException)`, trusting the unique index as the real guarantee and just preventing the race from crashing startup. Apply the same try/catch pattern to any future idempotent seeder.
- **Two real environment bugs were found and fixed, not hypothetical — expect to hit them again in any fresh environment:** (1) `postgres:18+` Docker images changed their volume-mount convention to `/var/lib/postgresql` (not `.../data`) — mounting at the old path crash-loops the container; `docker-compose.yml` already has the fix. (2) A JVM whose OS-detected default timezone resolves to the deprecated alias `Asia/Calcutta` gets rejected outright by this Postgres image at connection time — `ExpenseTrackerApiApplication` now pins `TimeZone.setDefault(Asia/Kolkata)` in a **static initializer** (not `main()`, since `@SpringBootTest` never calls `main()`), with a matching Surefire `argLine` in `pom.xml` for test JVMs. Any new entry point (e.g. a future CLI tool, a second `main()`) must not bypass this.
- **Spring Boot 4.1 has confirmed real breaking changes from older Spring Boot conventions** — verified empirically against the actual resolved jars, not assumed: dependency artifacts are now granular (`spring-boot-starter-webmvc`/`-webmvc-test`, no more monolithic `-web`/`spring-boot-starter-test`); `@MockBean` → `@MockitoBean` (`org.springframework.test.context.bean.override.mockito.MockitoBean`); `@WebMvcTest`/`@AutoConfigureMockMvc` now live under `org.springframework.boot.webmvc.test.autoconfigure`. Don't write Spring Boot 3.x-era test code here without checking the actual jars first — training-data assumptions about this stack are stale.
- **`GET /api/categories` already exists** (`CategoryController` → `CategoryService` → `CategoryRepository`, read-only) — Story 1.2's category selector should call this, not build a new endpoint. Full CRUD (create/rename/delete) is still Epic 2's job.
- **Category icon values seeded so far (`"food"`, `"transport"`, `"help-circle"`, etc.) are throwaway placeholders**, not a real icon set — the actual icon set is an explicitly deferred architecture decision, to be resolved when Epic 2 builds the Categories screen.
- **AC7 (actual Vercel/Railway deployment) was deliberately left undone by the dev agent** — provisioning real cloud accounts isn't something an AI dev agent should do autonomously (no credentials, and it's the project owner's infrastructure to own). `docs/deployment.md` has the exact manual steps, including the `SPRING_DATASOURCE_URL`/`USERNAME`/`PASSWORD` env vars needed since Railway's raw `DATABASE_URL` isn't a JDBC URL. The user explicitly agreed to move the story to `review` with this flagged rather than block on it — expect this to still be open unless someone reports back that they've done it.
- **No backend linter/formatter (Checkstyle, Spotless, etc.) was configured** — CI's backend job is `mvn test` only. Backend "lint" in later stories' ACs should either mean the same thing, or a future story should deliberately add a formatter rather than assuming one already exists.

## Story 1.2 Implemented (2026-07-05, read before touching `apps/web`'s Button/Link usage, Vitest config, or any future story's disabled-control styling)

Manual Transaction Entry is live and verified end to end via a real Playwright browser session against the real backend (not mocks): `1-2-manual-transaction-entry` is `review` in `sprint-status.yaml`. - **shadcn's `Button` component (`@base-ui/react/button`) is polymorphic via a `render` prop, not Radix's `asChild`.** `<Button render={<Link .../>} />` works, but `nativeButton` defaults to `true` and Base UI logs a dev-console warning if the rendered element isn't a real `<button>`. Setting `nativeButton={false}` silences the warning but **also overrides the element's accessible role to `"button"`** — wrong for anything that's actually real navigation (an `<a href>`). For a plain styled link (as opposed to a widget that merely looks like a button), skip the `Button`/`render` composition entirely and apply `buttonVariants()` directly to the target element instead — this was only caught by reading the live page's rendered HTML, not from the component code.
- **`DESIGN.md`'s "disabled is a real distinct color pair (`#E5E7EB`/`#9CA3AF`), not just low-opacity" instruction is easy to violate by default** — `opacity-50` on the enabled color reads as "disabled enough" in code review but was visibly wrong once screenshotted. Any future disabled-button styling (Epic 2/3, Story 1.5/1.6) should use the literal disabled hex pair, not an opacity shortcut.
- **Playwright's `.click()` refuses elements with `aria-disabled="true"` by default** (its own actionability model, not real browser/AT behavior) — use `.click({ force: true })` to test the "tap while aria-disabled, verify focus/no-op behavior" pattern this project uses for accessible disabled controls (established in Story 1.2 for the Save button, see `aria-describedby` note below). Don't mistake this Playwright friction for an app bug in a future e2e-test-writing story.
- **`@testing-library/react`'s auto-cleanup between tests requires a global `afterEach`** (e.g. Vitest's `globals: true`) to self-register — this project imports test APIs explicitly instead, so `apps/web/vitest.setup.ts` now calls `afterEach(cleanup)` itself. Any new test file relies on this shared setup; don't assume RTL cleans up on its own here.

Don't "helpfully" bump this dependency without checking that conflict is actually resolved upstream first.
- **`aria-describedby` is the established convention for disabled-control reasons** (confirmed working, not just planned — see Story 1.2 Created section above); the Save button uses `aria-disabled` (not native `disabled`) specifically so a tap while invalid can still move focus into the empty description field, per AC4. ## Story 1.2 Code-Reviewed and Done (2026-07-05, read before any future story that touches money/amounts, or Quick Add's UI)

Full adversarial review (Blind Hunter + Edge Case Hunter + Acceptance Auditor) ran against Story 1.2's diff; 1 decision-needed + 9 patch findings, all resolved and re-verified live same-day. `1-2-manual-transaction-entry` is `done`. Key project-wide precedents set, not just this-story fixes:

- **Money now travels as a JSON string end to end, not a JS `number` — this is the established project-wide convention, not just a Story 1.2 fix.** User explicitly decided this when a code-review finding noted AD-7's "`BigDecimal`/`NUMERIC(12,2)`, never `float`, end to end" claim was only true on the Java side. `TransactionDto.amount` uses `@JsonFormat(shape = JsonFormat.Shape.STRING)`; `CreateTransactionRequest.amount` needs no such annotation since Jackson already deserializes a quoted JSON string into `BigDecimal` by default — only the *response* shape needed forcing. **Every future money-touching story (Budget Settings 1.5, Dashboard totals 1.4/1.6, Search & Filter running total 3.1) must follow this same string-typed-money pattern on both the DTO and the `apps/web` TypeScript types**, not reintroduce a plain `number`.
- **`aria-disabled` (not native `disabled`) on a button that must still respond to taps needs a synchronous ref-based double-click guard, not just state.** `isSaving` (via `useState`) alone doesn't close a rapid-double-tap race, since a second click can fire before React commits the re-render — a plain `useRef` boolean checked and set synchronously at the top of the handler is what actually closes it. Any future conditionally-disabled-but-tappable control (same `aria-disabled` pattern as Quick Add's Save button) needs this same ref guard.
- **A native `<input type="date">`'s cleared `onChange` value is `""`, not `null`** — `pickedDate ?? undefined` doesn't catch this since nullish coalescing ignores empty strings, so a cleared date field was silently sending an empty string to the server. Normalize with `e.target.value || null` at the point of capture, not downstream.
- **A malformed/unparsable value in a JSON request body (bad enum, bad date, non-numeric string) throws `HttpMessageNotReadableException` during Jackson deserialization, *before* `@Valid` ever runs** — this is a different exception type than `MethodArgumentNotValidException` and needs its own `@ControllerAdvice` handler, or it falls through to the generic 500 catch-all. `GlobalExceptionHandler` now handles both; any future DTO with a non-trivial field type (dates, enums, nested objects) should assume this gap exists unless explicitly tested.
- **`DESIGN.md`'s "relative units (`rem`), not fixed `px`" instruction for typography is easy to violate with Tailwind arbitrary-value classes** (`text-[40px]` compiles to a literal fixed px, not scaling with browser zoom) — use `text-[2.5rem]`-style rem arbitrary values instead, and prefer an existing spec'd type-scale role (`label` 12px, `body` 14px, `heading` 17px, `amount-display` 40px) over inventing a new size.
- **A `Promise.all` over two independent fetches (one essential, one nice-to-have) fails the whole load if only the nice-to-have one errors.** Quick Add's category list (essential) and last-used-category (a pre-selection nicety) are now fetched independently — a `last-category` failure falls back silently to the first category rather than blocking the form. Apply the same "don't let an optional enhancement block the essential path" shape to any future similar combined-fetch.
- **Two false-positive review findings, confirmed wrong via direct technical analysis, not just dismissed on vibes:** (1) Hibernate's lazy `@ManyToOne` proxy's `.getId()` never triggers a `LazyInitializationException` regardless of session state, since the FK column value is already the identifier at proxy-construction time — a bare `.getId()` access is always safe. (2) `GET /api/categories` returning `[]` is structurally impossible per AD-5's SYSTEM-row partial-unique-index guarantee (at least one category always exists, undeletable) — don't re-flag either of these in a future review without new evidence.
