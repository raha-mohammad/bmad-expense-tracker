# Accessibility Review (WCAG 2.1 AA) — bmad-expense-tracker

## Overall verdict
The three budget-status color pairs are well-chosen — each uses a deliberately dark, near-black-value hue (green/amber/red) on a barely-tinted background, which reads as comfortably AA-safe by inspection even without a calculator. Color-alone reliance is genuinely avoided for budget severity in both the spec prose and the mockup markup. The gaps are elsewhere: one real contrast failure exists in the reference mockup's helper text, the focus-ring behavior is asserted but never actually specified with a concrete value, and a handful of plausible WCAG 2.1 AA success criteria (resize text, status messages, error identification for category naming) go unmentioned.

## 1. Budget-status color contrast — adequate (plausibility read)
- **Green** `#065F46` on `#ECFDF5` (DESIGN.md:19-20) — dark forest green on a near-white mint tint. Both channels are far apart in lightness; plausibly ~7:1+. Low risk.
- **Amber** `#92400E` on `#FFFBEB` (DESIGN.md:21-22) — this is a dark burnt-brown, not a bright/saturated amber, so it behaves like the green/red pair rather than a risky mid-tone amber (e.g. `#F59E0B` would have failed). Plausibly ~7:1+. Low risk.
- **Red** `#991B1B` on `#FEF2F2` (DESIGN.md:23-24) — dark brick red on a near-white pink tint. Plausibly ~7:1+. Low risk.
- All three pairs follow the same "dark-800-ish text on 50-ish tint" pattern, which is the standard accessible way to build a severity palette — no findings here, called out as a strength.

### Findings
- **[low]** No pair has an actual computed/verified ratio recorded anywhere (see review-rubric.md Finding 2) — the plausibility read above is not a substitute for a real check once real fonts/weights are chosen. Fix: run all three through a contrast checker once implemented and record the ratios in DESIGN.md.

## 2. Color-alone reliance — strong, with one watch item
- Budget severity: DESIGN.md:98 states "Every status color always ships paired with an icon and factual copy" and the mockup implements this exactly — `✅`/`▲`/`✕` (aria-hidden, decorative) alongside real text ("₹3,550 of ₹8,000 this month", "Budget exceeded by ₹250") at mockup:123,130-131. Spec and implementation agree. Strong.
- Frequent-expense chip vs. category button: DESIGN.md:129 explicitly requires a shape difference (pill vs. rounded-square), not just a color difference — correctly avoids color-alone. Strong.
- **[low]** Category icon button's selected state (DESIGN.md:128, mockup:159 `.cat-btn.sel`) is differentiated only by a border-color change (transparent → `{colors.primary}`) plus a background tint — no icon/checkmark/weight change accompanies it visually. The border's mere presence (not just its hue) likely satisfies WCAG 1.4.1 for most users, and `aria-pressed` (mockup:159-163) correctly carries the state to screen readers, but neither spec file states a non-hue visual cue as a requirement. Fix: consider requiring a small explicit non-color marker (e.g., a checkmark badge) given this is flagged in `.memlog.md` as the product's known drop-off point, or explicitly document why border+tint is judged sufficient.

## 3. Focus states, semantic HTML, disabled-reason detail — thin on focus, adequate elsewhere
- **Semantic HTML** — EXPERIENCE.md:103 is concrete enough to implement: "real `<button>`, `<label>`, `<input>`, and heading levels that reflect IA structure — not `<div onclick>` patterns." Adequate.
- **Disabled-button reason** — EXPERIENCE.md:105 specifies a concrete mechanism (`aria-describedby` pointing at "Add a description to save"). Adequate detail, though note the mockup itself implements the equivalent via `aria-label` instead (mockup:180) — a developer copying the mockup verbatim would diverge from the prescribed pattern (see review-rubric.md mechanical notes).
- **[high]** Focus ring is asserted as a goal but never actually specified. EXPERIENCE.md:88 and :102 both require a "visible focus ring... never suppressed," but as established in the rubric review, DESIGN.md never defines what that ring actually is (color, width, offset) — `ring` is explicitly called out as inherited-unmodified from shadcn (DESIGN.md:7-10), not primary-tinted. The only concrete values anywhere are in the mockup's CSS (`outline:3px solid var(--primary); outline-offset:2px`, mockup:92), which is illustrative, not spec. A developer working from the spines alone has no committed ring value to implement. Fix: commit the actual ring token (color/width/offset) in DESIGN.md.

## 4. WCAG 2.1 AA success criteria not mentioned
- **[medium] 4.1.3 Status Messages** — Inline states that appear without a page/route change and without moving focus (the "Couldn't save — check your connection" message at EXPERIENCE.md:85, the budget-invalid inline message at EXPERIENCE.md:172, and the Search & Filter running total updating live at EXPERIENCE.md:188) have no stated mechanism (e.g. `aria-live`) for announcing themselves to screen-reader users who aren't focused on that region. Fix: specify `aria-live="polite"` (or equivalent) for these inline status/error regions.
- **[medium] 1.4.4 Resize Text** — All typography is specified in fixed `px` values (DESIGN.md:28-45) with no statement about behavior at 200% browser zoom / OS text-scaling. For a manual-entry, numeric-heavy app this is a plausible real-world need. Fix: add a line noting text scales via relative units (`rem`) and content reflows without loss at 200%.
- **[medium] 3.3.1 Error Identification** — Covered for description-required (EXPERIENCE.md:82,105) and invalid budget amount (EXPERIENCE.md:172), but not for category-name entry (empty/duplicate name) — see review-rubric.md Finding under State coverage. Fix: define the error state and message.
- **[low] 2.5.5 Target Size vs. framing** — EXPERIENCE.md:106 and DESIGN.md:128 both cite "44×44px WCAG target size" as if it's an AA floor; in WCAG 2.1 the numeric Target Size criterion (2.5.5) is actually Level AAA, not AA (2.1 has no AA-level minimum target size at all). Not a functional problem — 60×60px comfortably clears it regardless — but the citation slightly overstates what 2.1 AA requires. Fix: reword as "exceeds common target-size guidance" rather than implying it's an AA requirement.
- **[low] 1.4.10 Reflow** — Responsive breakpoints are defined (EXPERIENCE.md:112-118) but there's no explicit statement that content reflows to a single column without two-dimensional scrolling at 320 CSS px / 400% zoom. Likely satisfied in practice given the mobile-first single-column design, but not stated as a requirement.
