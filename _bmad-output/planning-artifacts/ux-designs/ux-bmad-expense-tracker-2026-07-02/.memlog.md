- (decision by user) Primary users: students and young professionals who want simple manual daily expense tracking without traditional finance app complexity
- (decision by user) Desired feel: simple, minimal, fast, calm, non-judgmental, low-friction, easy to understand for first-time users
- (decision by user) Primary flows ranked: 1) Add Expense 2) View Dashboard 3) Set Monthly Budget 4) Manage Categories 5) Search and Filter Expenses
- (decision by user) Platform constraints: no auth in V1 (single-user app), no bank integration/receipt scanning/notifications, accessibility required (keyboard nav, sufficient color contrast, clear labels)
- (decision by user) Scope explicitly MVP-focused per user instruction
- (decision) Default categories (5, full CRUD): Food, Transport, Shopping, Bills, Entertainment - shallow taxonomy is deliberate (competitors' 15-20+ categories cause friction)
- (decision) Quick-add pattern: persistent floating + button, single-screen entry, numeric-keypad-first amount, today prefilled, last-used category as big icon button (tap to change), checkmark to save, target under 3 taps / under 5 seconds
- (decision) Frequent-expenses shelf: horizontal chips (e.g. 'Coffee $5') above/alongside keypad on the quick-add screen; tap logs immediately with today's date, zero further input
- (assumption) Frequent-expenses shelf ordering/cap (most-recent vs most-frequent, item count limit) is explicitly undecided in research - open question for this UX pass
- (decision) Budget status: traffic-light progress bar above the fold on dashboard home; honest-but-non-moralizing copy pattern, e.g. '$340 of $800 this month - on pace, 12 days left' / 'about $12 over this month'; detailed table via scroll, not forced upfront
- (decision) Category selection uses big icon buttons, never dropdowns (visual recognition over reading a list; category selection is the known drop-off point)
- (decision) Explicit anti-patterns to avoid in UI: red banners/alarm colors, pass/fail language, 'you have N unlogged days' counters, backlog view, scolding notifications
- (decision) Lapse/reopen state: reopening after a gap shows only today's picture, no confrontation UI, no catch-up burden
- (decision) Primary research persona: 'The Anxious Starter', ~18-24, first time managing money, no budgeting experience, zero patience for setup - primary lens for V1 journeys
- (decision) Trust microcopy considered: explicit 'we never connect to your bank' reassurance in onboarding/settings; manual entry framed positively ('you're always in control'), not as a limitation
- (decision) Rejected/deferred ideas not to reintroduce in this pass: home-screen widget, chill/strict budget tone toggle, voice-to-log, weekly-only check-in cadence, freeform quick-capture parsing (H2 future), lapse-recovery re-entry screen (gated behind untested hypothesis)
- (gap) No accessibility (WCAG/screen reader/keyboard nav specifics), typography, color, or responsive-breakpoint guidance exists in any source doc - genuinely open for this UX pass despite accessibility being a stated requirement

- (override by user) OVERRIDES market-research anti-pattern 'no red banners': user wants explicit green/amber/red traffic-light budget status, including a visibly red state when budget is exceeded. Non-shaming spirit preserved in COPY tone (factual/informative, e.g. 'Budget exceeded by ₹250', 'You've spent 105% of your monthly budget'), not in color choice. Distinction: color signals severity, copy stays neutral/factual, never moralizing ('you failed', 'you're bad with money').
- (decision by user) Currency: ₹ INR throughout the app (supersedes $ used in research docs and earlier journey draft)
- (decision by user) Budget status thresholds: green <80% of budget, amber 80-100%, red >100% (exceeded)

- (decision by coach) Category deletion: the 5 default categories (Food, Transport, Shopping, Bills, Entertainment) are non-deletable, rename/re-icon only. User-added custom categories are fully deletable. Rationale: keeps the deliberately shallow default taxonomy from research intact while still allowing personalization.
- (decision by coach) Deleting a category with existing transactions: auto-reassign those transactions to an implicit read-only 'Uncategorized' bucket rather than blocking the delete or forcing manual reassignment. Rationale: blocking/forcing cleanup contradicts the product's low-friction, never-burden-the-user philosophy; 'Uncategorized' still surfaces in dashboard totals and search/filter.

- (decision by user) i18n: English only for MVP
- (decision by user) Number formatting: Indian digit grouping for currency (e.g. ₹1,00,000), not international thousands-grouping
- (decision by user) Offline: always-online assumption, no offline queueing/sync; a failed save shows a plain connection-lost state only
- (decision by user) Motion: minimal, near-instant transitions, no decorative animation - reinforces the 'instant update' feel over a 'calm micro-animation' feel
