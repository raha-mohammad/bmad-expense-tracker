- (constraint) Monolith: Next.js frontend + Spring Boot backend + PostgreSQL, REST/JSON only between frontend and backend; frontend never touches Postgres directly. [ADOPTED from technical research, PRD addendum]
- (constraint) Controller -> Service -> Repository layering mandatory even for simple endpoints. [ADOPTED, PRD addendum]
- (constraint) DTOs at the API boundary always; controllers never return JPA entities directly. [ADOPTED, PRD addendum]
- (constraint) Centralized exception handling via @ControllerAdvice, configured early not bolted on. [ADOPTED, PRD addendum]
- (constraint) CORS must be configured on Spring Boot before the first frontend call. [ADOPTED, PRD addendum]
- (constraint) Derived-totals invariant is structural: no totals/remaining/spent column may exist anywhere in the schema; every category/period/budget total is computed live from Transaction rows at read time. Non-negotiable domain rule (PRD FR-5, NFR, addendum). [ADOPTED]
- (constraint) No auth/accounts in v1, single implicit user, backend-persisted (not local storage). Stateless JWT is the named future branch point if accounts are ever added -- not evaluated further now. [ADOPTED]
- (constraint) Currency is INR only, Indian digit grouping, no multi-currency. Always-online assumption, no offline queueing/sync; failed saves show inline retry-capable error, values retained. [ADOPTED]
- (constraint) Basic @WebMvcTest controller tests (status -> content-type -> payload) are sufficient for MVP scope; no heavier test-framework investment unless complexity grows. [ADOPTED, PRD addendum, guidance not hard rule]

- (decision by user) Hosting: split, managed. Next.js frontend on Vercel (free tier). Spring Boot backend + PostgreSQL together on Railway (Hobby $5/mo, persistent volumes, no expiry/sleep). Chosen over single-VPS (more ops burden) and Render-paid (pricier, needs separate frontend deploy) because it minimizes ops for a solo builder while satisfying the PRD's non-ephemeral persistence NFR. Render free tier explicitly ruled out: free Postgres expires after 30 days, violating the NFR directly.
- (decision by user) Repo topology: monorepo (single existing git repo), apps/web (Next.js) + apps/api (Spring Boot), each platform (Vercel/Railway) configured to deploy from its subfolder.
- (decision by user) CI/CD: GitHub Actions gate from day one -- workflow runs backend + frontend tests/lint on every push/PR before merge, ahead of platform-native auto-deploy. Chosen over deploy-only for portfolio credibility and early regression-catching, despite solo/personal-use context.
- (decision by user) Budget carry-forward: when a new Period starts, its Budget amount auto-copies from the prior Period's amount (still independently editable per-period, spend still starts at 0). Resolves an ambiguity the PRD left open (rollover of spend vs. the budget figure itself). Chosen to match the product's low-friction ethos over forcing monthly re-entry.
- (decision) Budget data model: one budget row per Period (period_start, amount), not a single mutable value -- makes 'no rollover of spend' and 'carry-forward of amount' both structural (each period's spend total is independently derived; the amount is a copy, not a live reference), and supports future per-period budget history without a schema change.
- (decision by user) Category model: 'kind' distinguishes DEFAULT (5 seeded, renameable/re-iconable, never deletable), CUSTOM (user-created, fully editable+deletable), SYSTEM (Uncategorized: real seeded singleton row, not a nullable FK sentinel, not shown in any edit/delete UI). Deletion reassignment is a uniform FK update to the SYSTEM row, never special-cased null-handling.
- (decision by user) IDs: auto-increment BIGINT primary keys, not UUID -- simplest fit for a single-instance solo app at this scale.
- (decision by user) API shape: resource-oriented REST under /api/* (/api/transactions, /api/categories, /api/budget), no /v1 versioning yet (YAGNI). Consistent {error:{code,message}} envelope for failures via @ControllerAdvice.
- (decision by user) Money/dates: server-side BigDecimal / NUMERIC(12,2) for all amounts, never float. ISO-8601 dates. Client only ever displays totals the server computed -- never sums client-side, which is what keeps the derived-totals invariant real end-to-end, not just in the DB.
- (decision by user) Testing shape: backend @WebMvcTest per addendum guidance, plus direct unit tests on the derived-totals/budget-status calculation specifically (the core integrity guarantee). Frontend component tests (Vitest/Testing Library) on Quick Add and Budget Status logic. No e2e for MVP.
